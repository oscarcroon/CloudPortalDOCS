# API tokens

API keys give programs and integrations access to your organization's data via the portal API — for example for automated workflows, CI/CD or external systems. Each key only gets the permissions you choose.

::note
To create or revoke API keys you need permission to manage the organization. If you lack access, an information box is shown and the buttons are disabled.
::

## Overview of keys

For each key, the list shows:

- **Prefix** — a short, recognizable beginning of the key (the full key is never shown again after creation).
- **Description** — your own text describing what the key is used for.
- **Permissions (scopes)** — what the key is allowed to do.
- **Created** and who created it.
- **Last used**.
- **Status** — Active, Expired or Revoked.

## Creating an API key

1. Click **Create new API key**.
2. Enter a **description** (optional but recommended, for example "CI/CD pipeline" or "External integration").
3. Choose **permissions (scopes)**. You can tick individual permissions or use one of the quick templates to fill in a set at once.
4. Set an **expiration date** (optional). If you leave the field empty, a key without an expiration date is created.
5. Click **Create key**.

::warning
The key is shown **only once**, right after it has been created. Copy it and store it in a safe place before you close the box. If you lose it, you must create a new one.
::

::tip
Give each integration its own key with only the permissions it needs, and a descriptive text. That makes it easy to see what a key is used for and to revoke the right key if something needs to be withdrawn.
::

## Using the key

Include the key in the request's authentication header:

```
Authorization: Bearer <your-key>
```

## Revoking an API key

1. Find the key in the list.
2. Click **Revoke** on the key's row.
3. Confirm in the dialog box.

::warning
Revocation cannot be undone, and a revoked key cannot be reactivated. All integrations that use the key stop working immediately — update them with a new key before you revoke the old one.
::

## Good practices

- Set an **expiration date** on keys that are only needed for a limited time.
- **Revoke** keys that are no longer in use.
- Never share a key in email, chat or code that is checked into a repo.
- Use the **Last used** column to spot keys that can be removed.
