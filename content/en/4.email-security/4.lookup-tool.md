# Lookup tool

The lookup tool is a troubleshooting tool for DNS and email security. You can look up individual DNS records or run a combined quick check of a domain. The tool reads published records — it changes nothing.

You'll find it under **Email security → Lookup**. The tool has two tabs: **DNS lookup** and **Mail Security Quick Check**.

## DNS lookup

Use this tab to see the raw DNS records for a domain or hostname.

1. Enter a **domain or hostname** (e.g. `customer.com`).
2. Select which **record types** you want to look up by clicking the tags (e.g. MX, TXT, A, AAAA). Your selection is saved for next time.
3. Click **Look up**.

The result is shown per record type. For records such as MX and SOA, relevant columns are shown (e.g. priority and MX host, or serial and TTL values). You can copy individual values, and rerun the lookup with **Rerun**.

::tip
If there is no match for a selected type, this is clearly shown per type — it means the domain has no record of that kind.
::

## Mail Security Quick Check

This tab runs a combined check of the domain's email security and shows an overall status: **OK**, **Warning** or **Error**.

1. Enter the **domain**.
2. Click **Run check**.

The check looks at, among other things:

- **MX** — which mail servers receive email (including implicit fallback and null MX).
- **SPF** — whether an SPF record exists, whether there are several, and an estimated lookup count against the limit of 10.
- **DMARC** — whether a DMARC record exists at `_dmarc.<domain>` and which policy applies.
- **MTA-STS** — discovery record, policy file and mode.
- **TLS-RPT** — report URIs at `_smtp._tls.<domain>`.
- **MTA-STS policy host** — that the policy file can be fetched over HTTPS.
- **DNSSEC** — best-effort signal of whether the domain is signed.
- **CAA** — which certificate authorities are allowed.
- **BIMI** — whether a BIMI record is configured and its logo and certificate URLs.

Each check gets its own status, and warnings explain what should be addressed.

::note
If a domain publishes a null MX record (it explicitly does not receive email), the mail-related checks are skipped. DNSSEC and CAA are still relevant for the domain itself.
::

::note
Some checks are "best-effort" signals. The MTA-STS policy host check, for example, only confirms that the HTTPS handshake succeeded — it does not inspect the certificate's issuer or expiry date. The DNSSEC check only looks at whether DNSKEY exists, not at full validation.
::

## Jump to DNS lookup

From a card in Mail Security Quick Check you can jump directly to **DNS lookup** with the right name and record type prefilled, so you can quickly inspect the underlying record.
