# Import app

With the import wizard you bring in an app you built in Lovable, Cursor, Claude Code, Bolt or any Git repo and prepare it for hosting. Your code stays where it is — the wizard analyzes the repo and builds a plan.

::note
The import prepares the hosting environment and creates a plan. It does not change your build tool, your repo or any database you may have.
::

## Before you start

- First connect your GitHub account. See [Connect GitHub](/en/app-hosting/connect-github).
- Make sure your latest code is pushed to the repo.
- If you build in Lovable: connect the project to GitHub and confirm that the latest version is synced.

## Start the import

1. Go to **App Hosting** and click **Import application**.
2. Follow the wizard step by step. The **Next** button takes you forward and **Back** one step back.

### Step 1 – Where is the app built?

Choose the tool you built the app in (Lovable, Cursor, Claude Code, Bolt or a plain Git repo). Your choice tailors the following steps. For Lovable and Git, a short guide is shown for how to retrieve the repo's URL.

### Step 2 – Connect the repo

Choose how App Hosting should reach the repo:

- **GitHub (recommended)** — select the repo from a list. The app only sees the repos you have granted it access to. If a repo is missing from the list, you can select more in GitHub and then click **Refresh**.
- **Private with GitHub App** — if you already have an installation, enter its installation ID.
- **Advanced** — enter the repo's URL directly and add an access token for private repos, or specify a public repo.

Also enter the **branch** (the branch to be built) and an **application name**.

### Step 3 – Analyze the app

App Hosting analyzes the repo and automatically identifies the framework and build settings. Review the result.

### Step 4 – Confirm framework / build

Check and correct the detected settings — they control how the app is built:

- **Framework**
- **Build command**
- **Runtime**
- **Package manager**

::tip
If the package manager is uncertain, look for a lockfile in the repo: `package-lock.json` = npm, `pnpm-lock.yaml` = pnpm, `yarn.lock` = yarn, `bun.lockb` = bun.
::

Check **I have reviewed the settings** to proceed.

### Step 5 – Secrets / environment variables

Add any environment variables the app needs. This is recommended but not required here — you can complete it later. Read more in [Secrets and environments](/en/app-hosting/secrets-and-environments).

::warning
Values that start with `VITE_` or `NEXT_PUBLIC_` are built into the finished app and become public in the browser. Never put a secret server key there.
::

### Step 6 – Backend / database

Choose where the production database should run. For many apps, a separate Supabase Cloud project for production is the simplest and recommended choice. If you are unsure, you can ask CoreIT to review.

### Step 7 – Environments

Choose which environments to create:

- **Test + Production** — lets you keep building without testing directly in live operation.
- **Production only** — a single environment. A test environment can be added later.

### Step 8 – Hosting environment and size

Choose where the app should be hosted among the environments enabled for you (for example with CoreIT or in your own Azure). Then choose a size; the portal can suggest a recommended size based on the analysis.

::note
If you see no hosting environments, an administrator needs to enable one for you. Come back when it is in place.
::

### Step 9 – Plan and finish

Before anything is created, a preview is shown of what will be created — application, environment, hosting environment, size, secrets and backend. Review the plan and click **Create application** to complete the import.

## After the import

The app appears under **My apps**. From there you review the analysis, adjust the build configuration and proceed to building and publishing.

Continue developing in your tool as usual. When the repo is updated, you can run **Analyze again** on the app's page to capture the latest version.
