# Deploy app

Once you have imported an app, you prepare it for building and publishing on the app's page in App Hosting. Here you choose the deploy branch, confirm the build configuration, choose a delivery method and follow the build.

## Prepare the app for building

Open the app under **My apps** and go through the following panels.

### Build configuration

Check the detected build settings (framework, build command, runtime and package manager). Your choices control both the plan and the generated workflow file. Changes update the plan preview and the workflow below. Save with **Save**, or **Reset to detected** to revert to what was analyzed.

### Deploy branch

Specify the branch to be deployed. The workflow triggers on push to this branch. For GitHub repos you can select the branch from a list; otherwise you type the branch name by hand.

::note
When you switch branch, the analysis is rerun on the new branch.
::

### Plan / preview

The **Plan / preview** panel shows what will be created before anything is committed — frontend, any backend, delivery method, environments and which secrets are used (names only, never values).

## Choose a delivery method (CI/CD)

Under **CI/CD — delivery method** you choose how releases reach the environment:

- **Manual deploy from the portal**
- **GitHub Actions per release**
- **GitHub Actions + manual deploy**

You can preview and download the workflow file before it is added:

1. Review the generated workflow.
2. Click **Download workflow.yml** if you want to add it manually, or **Commit to repo** to add it directly (GitHub repos with write access only).
3. Confirm with **Yes, commit**.

::tip
If you need to customize the workflow, choose **Customize workflow**, edit freely and save. A customized workflow is marked as **Customized**. You can always go back with **Reset to generated**.
::

::note
Committing the workflow directly from the portal is only supported for GitHub repos. For other repos, download the file and add it manually.
::

## Build and publish

For apps hosted on Azure Static Web Apps, the build itself runs in your GitHub Actions. The progress is shown in the portal and goes through these states:

1. **Azure Static Web App created**
2. **CI configured** (secrets + workflow)
3. **Build started in GitHub Actions**
4. **Building in GitHub Actions**
5. **Live**

When the build is complete, the app becomes available at its **Live URL**. If you use a pull request-based workflow, a step may show **Workflow in PR — waiting for merge**; the app goes live after the PR is merged.

::note
Screenshot: the app's page with the CI progress and the status label moving toward "Live".
::

## Status and operations

On the app's page you see continuous status, for example whether a new version exists in GitHub. Run **Analyze again** to capture the latest commit. Health checks can be configured to monitor the app while it runs.
