# Secrets and environments

Environment variables and secrets control how your app connects to databases and external services. In App Hosting you manage them in the import wizard's **Secrets / environment variables** step and later on the app's page.

::note
Adding values is recommended but not required to proceed — you can complete it later.
::

## Public versus server values

App Hosting distinguishes between two types of values:

- **Public / frontend** — values that start with, for example, `VITE_` or `NEXT_PUBLIC_`. They are built into the app at build time and are visible in the browser.
- **Server / runtime** — secrets that are only used on the server and never exposed in the frontend.

::warning
Frontend values are public in the finished app — never treat them as secrets. Only put secret keys among the server values.
::

## Add an environment variable

1. Open the **Environment variables** section (in the import wizard or on the app's page).
2. Click **Add secret**.
3. Enter a **Key** and a **Value**.
4. Choose the **scope** — public/frontend or server/runtime.
5. Save.

Missing values the app needs are clearly flagged so you can see what remains.

## Common variables

The portal provides guidance for common values. Some examples:

- **Supabase URL** — your Supabase project's API URL (Project Settings → API → Project URL).
- **Supabase anon/publishable key** — the public key, normally harmless in the frontend (Project Settings → API → anon/public).
- **Supabase service_role key** — full admin access. Server only, never in the frontend.
- **Database connection** — server only.
- **Stripe** — use the public publishable key in the frontend and the secret key only on the server.

::warning
The Supabase service_role key grants full access to your database. It must never be placed among the frontend values.
::

## Environments (test and production)

During import you choose whether the app should have both **Test + Production** or **Production only**. A test environment lets you keep building in your tool without testing directly in live operation. You can add a test environment later if you start with production only.

::note
Environment variables belong to a specific environment. Values you add for test do not affect production and vice versa.
::
