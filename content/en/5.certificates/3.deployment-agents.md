# Deployment and agents

An **agent** is a small background service that runs on your server. It receives certificate orders from the portal, handles validation, and installs the certificate locally. With an agent in place, the portal can deploy and renew certificates automatically – without you logging in to the server every time.

::note
Requirements for an agent: Windows Server 2016+ or Windows 10+, administrator rights on the server, and outbound HTTPS access to the portal.
::

## Register a new agent

1. Go to **Certificates → Agents**.
2. Click **Register agent**.
3. Fill in the form:
   - **Name** – a clear name, for example `web-server-01`.
   - **Description** (optional).
   - **Capabilities** – check what the server should be able to handle, for example HTTP-01, DNS-01, IIS, PFX, or Central SSL.
4. Click **Register agent**.

::warning
Immediately after registration, the agent's token is shown once. Copy and store it securely – it won't be shown again. If you need a new token later, use **Generate new token**, but the old one stops working immediately.
::

## Install the agent on the server

Once you've registered the agent, an **installation wizard** opens. You can also reach it later via the **Installation wizard** button on the agent's row in the list. The wizard guides you through the installation step by step:

1. **Agent token** – copy the token from the wizard. It's needed for the agent to connect to the portal.
2. **Download and install** – click **Download agent** to get the agent package, and **Download installation script** for the PowerShell script that automates the installation.
3. **Run the installation script** – extract the downloaded file and run the script in PowerShell **as administrator**. The script copies files, writes the configuration, creates the Windows service, and starts it. There's also a manual configuration mode for those who want to set everything up themselves.
4. **Verify** – the wizard shows a live indicator. Once the service is running, the agent should appear as **online** within about 60 seconds, with its last seen IP and timestamp.

::tip
The agent always connects outbound to the portal. Your servers don't need to be reachable from the outside – it's enough that they can reach the portal over HTTPS.
::

## View the status of your agents

On the **Certificates → Agents** page you'll see each agent with:

- Name and status (active/inactive).
- An **online** marker with a pulsing dot when the agent has checked in recently, otherwise **offline**.
- Last heartbeat and last seen IP.
- The agent's capabilities shown as tags.

An agent counts as online when it has checked in during the last few minutes. If it doesn't check in, it's marked as offline.

## Deploy certificates with an agent

When you order a certificate, you choose which agent should carry it out and which installation target applies (see [Order a certificate](/en/certificates/order-certificate)). The agent then installs the certificate in the right place and reports back to the portal.

You can use the same certificate on multiple servers – each installation is tracked separately, so the portal can show where the certificate is actually deployed and whether anything has changed. Such discrepancies are tracked under [Discovery and inventory](/en/certificates/discovery-and-inventory).

::note
The portal asks the agent to install the certificate, but it's the server and its configuration that determine whether the certificate is actually put into use. Check the service on the server if an installation isn't showing up in the portal.
::

## Troubleshooting

| Problem | What to check |
|---------|-----------------|
| The agent doesn't appear as online | That the CertAgent service is running on the server. |
| The agent can't reach the portal | That the server can reach the portal over outbound HTTPS. |
| The connection is refused | That the correct agent token is configured. Generate a new token if needed. |
| The installation gives an unexpected error | The agent's logs on the server. |
