# Deploya app

När du importerat en app förbereder du den för bygge och publicering på appens sida i App Hosting. Här väljer du deploy-gren, bekräftar bygg-konfigurationen, väljer leveransmetod och följer bygget.

## Förbered appen för bygge

Öppna appen under **Mina appar** och gå igenom följande paneler.

### Bygg-konfiguration

Kontrollera de detekterade bygg-inställningarna (framework, build-kommando, runtime och pakethanterare). Dina val styr både planen och den genererade workflow-filen. Ändringar uppdaterar plan-förhandsvisningen och workflow nedan. Spara med **Spara**, eller **Återställ till detekterat** för att gå tillbaka till det som analyserades.

### Deploy-gren

Ange grenen som ska deployas. Workflow:en triggar på push till den här grenen. För GitHub-repos kan du välja gren ur en lista; annars skriver du grennamnet för hand.

::note
När du byter gren körs genomlysningen om på den nya grenen.
::

### Plan / förhandsvisning

Panelen **Plan / förhandsvisning** visar vad som kommer att skapas innan något verkställs — frontend, eventuell backend, leveransmetod, miljöer och vilka secrets som används (endast namn, aldrig värden).

## Välj leveransmetod (CI/CD)

Under **CI/CD — leveransmetod** väljer du hur releaser når miljön:

- **Manuell deploy från portalen**
- **GitHub Actions per release**
- **GitHub Actions + manuell deploy**

Du kan förhandsvisa och hämta workflow-filen innan den läggs in:

1. Granska den genererade workflow:en.
2. Klicka **Ladda ner workflow.yml** om du vill lägga till den manuellt, eller **Committa till repo** för att lägga in den direkt (endast GitHub-repos med skrivåtkomst).
3. Bekräfta med **Ja, committa**.

::tip
Behöver du anpassa workflow:en väljer du **Anpassa workflow**, redigerar fritt och sparar. En anpassad workflow märks som **Anpassad**. Du kan alltid gå tillbaka med **Återställ till genererad**.
::

::note
Att committa workflow direkt från portalen stöds bara för GitHub-repos. För andra repos laddar du ner filen och lägger till den manuellt.
::

## Bygg och publicera

För appar som hostas på Azure Static Web Apps körs själva bygget i din GitHub Actions. Förloppet visas i portalen och går igenom dessa lägen:

1. **Azure Static Web App skapad**
2. **CI konfigurerad** (secrets + workflow)
3. **Bygget startat i GitHub Actions**
4. **Bygger i GitHub Actions**
5. **Live**

När bygget är klart blir appen tillgänglig på sin **Live-URL**. Använder du en pull request-baserad workflow kan ett steg visa **Workflow i PR — väntar på merge**; appen blir live efter att PR:en mergeats.

::note
Skärmdump: appens sida med CI-förloppet och statusetiketten som rör sig mot "Live".
::

## Status och drift

På appens sida ser du löpande status, till exempel om en ny version finns i GitHub. Kör **Analysera igen** för att fånga den senaste commiten. Health checks kan konfigureras för att bevaka appen när den körs.
